## CRITICAL Findings — Action Required by Rusty / Basher

### 🔴 1. `azure-ai-projects` 2.x — MAJOR VERSION BREAK

The package jumped from `1.0.0b4` (beta) to `2.1.0` (GA). This is a **complete SDK rewrite**:

- **Import path:** `from azure.ai.projects import AIProjectClient` — **unchanged**, still works in 2.x ✅
- **Constructor:** Old 1.x beta used a connection string. New 2.x requires `endpoint=` kwarg:
  ```python
  # OLD (1.x beta) — may not work
  client = AIProjectClient.from_connection_string(conn_str, credential)
  
  # NEW (2.x) — required pattern
  client = AIProjectClient(endpoint=os.environ["FOUNDRY_PROJECT_ENDPOINT"], credential=credential)
  ```
- **Endpoint URL format changed:** Old: `https://<hub>.api.azureml.ms/...` → New: `https://<resource>.services.ai.azure.com/api/projects/<project>`
- **`.agents` sub-client** is now the primary way to run agents (replacing standalone Agent SDK)
- **`.get_openai_client()`** replaces the old `ChatCompletionsClient` wrapper
- The `.agents/skills/microsoft-foundry/finetuning/scripts/common.py` already uses the **new 2.x pattern** ✅

**Rusty/Basher action:** Scan all challenge code samples for `AIProjectClient.from_connection_string` and replace with `AIProjectClient(endpoint=..., credential=...)`. Also verify `AZURE_AI_ENDPOINT` env var format matches the new Foundry project endpoint URL pattern.

---

### 🔴 2. `azure-search-documents` 12.x — MAJOR VERSION BREAK

Bumped from `>=11.4.0` to `>=12.0.0`. Microsoft typically makes breaking changes across major Search SDK versions (index field types, query options, result shape).

**Rusty/Basher action:** Review any RAG / Challenge 04 code that constructs `SearchClient`, `SearchIndexClient`, `SearchIndex`, or uses `VectorizedQuery`. The vector search API surface changed significantly between 11.x and 12.x.

---

### 🟡 3. `openai` 2.x — MEDIUM RISK

Bumped from `>=1.35.0` to `>=2.0.0`. Key changes:
- `client.chat.completions.create()` **still works** (supported indefinitely)
- New preferred API: `client.responses.create()` (Responses API)
- `client.OpenAI()` and `client.AzureOpenAI()` constructors — **unchanged** ✅

**Rusty/Basher action:** Low priority; existing `chat.completions` code will keep working. Optionally update code samples to use `responses.create()` to align with current OpenAI guidance.

---

### 🟡 4. `azure-ai-inference` — Still Beta

`azure-ai-inference` is still at `1.0.0b9` (Development Status: 4 - Beta). No GA release has shipped as of 2026-05-28. The `from azure.ai.inference import ChatCompletionsClient` import is still correct.

**Note:** Microsoft documentation now strongly encourages using `azure-ai-projects 2.x` + `get_openai_client()` instead of `azure-ai-inference` directly. Any challenge samples that still use `ChatCompletionsClient` should flag this in their coach notes.

---

### 🟡 5. `promptflow` — Maintenance Mode Signal

`promptflow` is still published and active (1.18.5), but Microsoft's investment has shifted toward **Azure AI Foundry Agent Service** and the new **Foundry SDK flows**. Promptflow is not deprecated but is in a "sustaining" posture. The VS Code extension (`ms-toolsai.promptflow`) remains in devcontainer.json.

**Recommendation:** Keep `promptflow` in `requirements.txt` for Challenge 03 compatibility. Add a coach note to the challenge that newer Azure AI Foundry projects use the Agent Service or `azure-ai-projects` SDK flows rather than classic promptflow DAG flows.

---

### 🟡 6. `ipykernel` 7.x — MEDIUM RISK

Bumped from `>=6.29.0` to `>=7.0.0`. IPython kernel 7.x may have changes to the Jupyter messaging protocol or kernel startup. Low practical risk for standard notebook use, but worth smoke-testing if the hackathon has custom kernel configuration.
